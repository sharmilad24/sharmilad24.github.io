Five bugs that taught me more than the model did

July 2026 · Building a MARTA delay predictor

Last month I built a trip planner for MARTA, the Atlanta rail system. It collects live train data every two minutes, stores it in SQLite, and uses a Random Forest model to predict delays within 78 seconds, about 20 percent better than a simple baseline. There is a Streamlit app that routes real addresses through the rail network and a Power BI dashboard on top of the same pipeline.

That is the version that goes on a resume. This post is about the other version, the five problems that actually taught me something. None of them were machine learning problems. All of them were the kind of thing you only learn by shipping.

1. The file that existed but did not exist

Early on, Python kept telling me my data file did not exist. I could see the file. I could open it. Python insisted it was not there. This happened more than once before I understood what was going on.

My scripts were saving files relative to wherever I happened to run them from, and sometimes that meant everything landed inside my virtual environment folder instead of the project root. So the file existed, just not where the next script was looking.

The fix was boring and permanent: stop trusting relative paths. Anchor everything to the project root and build paths from there. I have not lost a file since.

What it taught me: when the computer says something impossible, one of your assumptions is wrong. The file exists and the file cannot be found were both true. The hidden assumption was that we agreed on where here was.

2. Database is locked, again

This was the bug that ate real time. My collector wrote fresh train data into SQLite every two minutes. That part worked great. But whenever the Streamlit app or any analysis script tried to read while the collector was writing, everything crashed with one short message: database is locked.

I went through the classic arc. First I added a longer timeout, which just made the crash slower. Then I tried enabling write-ahead logging, and the command to switch it on got blocked by the very lock I was trying to fix. The thing that finally held was opening every reader in true read-only mode so it never tried to take a write lock at all:

conn = sqlite3.connect("file:marta.db?mode=ro", uri=True)

One writer, many readers, nobody fighting over the lock.

What it taught me: concurrency problems do not care that your code is correct in isolation. Both my writer and my readers were fine alone. The bug lived in the space between them.

3. The Green Line was arriving early, apparently

My first average delay by line chart said the Green Line was running 9 minutes early. Trains do not run 9 minutes early. For a few minutes I wondered if I had discovered something. I had not. I had discovered a small sample.

Early in collection, the Green Line only had a handful of records, and a couple of odd readings dragged the whole average negative. Once I filtered out lines below a minimum record count, the nonsense disappeared and the chart started telling the truth.

What it taught me: small samples lie confidently. Now any aggregate I show comes with a sample size check first, because an average without one is just a rumor with an axis.

4. Every address in Georgia was apparently downtown Atlanta

The trip planner has a first and last mile feature that estimates the drive between your address and the nearest station. It shipped broken, and it was broken silently, which is the worst kind.

My geocoding code force-appended Atlanta, GA to every address to help the geocoder out. Which worked fine for Atlanta addresses and quietly relocated everyone else. Stone Mountain showed up as 0.8 miles from a station when it should have been well over 5. Nothing crashed. The numbers were just wrong.

I only caught it because I sanity-checked a trip I personally knew. The fix was to stop mangling the input and let the geocoder see the real address, with the search biased to Georgia instead of hard-coded to one city.

What it taught me: silent wrong answers are more dangerous than crashes. A crash files its own bug report. A wrong distance just sits there looking plausible. Test with inputs where you already know the answer.

5. The router that gave up at Five Points

The first version of my router only handled trips on a single line. Ask it for Georgia State to the Airport, one of the most common real trips in the city, and it returned nothing, because that trip needs a transfer at Five Points and my router did not know transfers existed.

Building transfer handling meant treating the network like an actual graph, where Five Points connects lines, rather than four separate lines that happen to share a map. Once transfers worked, routes that real riders actually take started resolving.

What it taught me: your first data model is usually a simplification of the real system, and the gap shows up exactly where users live. The single line assumption was invisible until the most normal trip in Atlanta broke it.

What I would tell someone starting a similar project

The model was honestly the easy part. Scikit-learn is mature, Random Forests are forgiving, and feature engineering on time series data is well documented. Almost all of my debugging time went into the unglamorous layer underneath: file paths, database locks, geocoding, and the difference between a network diagram and a real network.

If I could send one sentence back to myself at the start, it would be this: build the smallest end to end version first, one line, one station pair, one prediction, and only then make it bigger. Every one of these five bugs would have surfaced earlier and cheaper.

The trip planner is live at marta-trip-planner.streamlit.app and the rest of my work is at sharmilad24.github.io. If you are building something with transit data and hit a wall, my email is on the homepage, I have probably hit the same wall.
